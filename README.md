# 1. Introduction to the final project

Welcome to the Final Project.

For the next 7 weeks you will be focused on building a fullstack project and you will be presenting this project on the **19th February 2025**.

There will be a couple of phases of this project spread out in 4 sections:

-   Phase one: Idea creation (2-3 days)
-   Phase two: Designing and Planning (2-3 days)
-   Phase three: Implementation (18-20 days)
-   Phase four: Presentation (4* days) * This will be dependant on the other phases

**Expectations**: It is good to note at the beginning of the project what to expect. This will be very demanding experience, we as teachers are here to support and guide you _however_, a lot of the idea and integration will be set up on the side of the students. Two things very important to keep in mind during the early phases is:

-   **Do not jump straight into coding**, this will surely lead to a difficult outcome, perhaps the most crucial time to use is the first days in planning ideas and sketching out an MVP (Minimum viable product).
-   **Don't go overboard with features**, focus on a stable working version, then add features as you go.

### 2. Finding project ideas

For today, I would like all teams to come up with ideas. These ideas can be literally anything, try to think about what interests you and bring ideas to the table for your group. If coming up with an idea for your group proves difficult think of the following problems to help drive creativity:

-   Think about the things you do everyday. Is there something you could build that would help you with your everyday tasks?
-   Does somebody close to you struggle with a task in their everyday life? Is their something you could build that would help them?
-   Think on your hobbies and interest, is there something you can build that would help make them more enjoyable?

### 3. Team Structure

The team should share responsibilities but will define roles:

-   **Project Manager** - Responsible for team communication and co-ordination. The Project Manager should organize any team meetings and should ensure the group stays on track.
-   **Repository Maintainer** - Responsible for enforcing correct GIT processes, such as ensuring team members are not directly committing to the main / master branch, and that new features are added using the PR (pull request) system.
-   **Lead Frontend Developer** - Responsible for co-ordinating the front-end tasks and for providing support to the other frontend developers.
-   **Lead Backend Developer** - Responsible for co-ordinating the backend-end tasks and for providing support to the other backend developers.

This is a template for how groups should look, I would suggest at minimum having a Project Manager, someone leading the projects and assigning task is crucial for a stable and productive work environment.

### 4. Create feature list

-   What problem will you be solving with this application?
-   What can the user do with it?
-   Create MVP (minimum viable product) feature list project roadmap (what we want to achieve by the end of the 7 weeks if everything goes as planned)

Here it is a great idea to start with user stories: "A user story is an informal, general explanation of a software feature written from the perspective of the end user. Its purpose is to articulate how a software feature will provide value to the customer."

For example: **As a [persona], I [want to], [so that].**

-   this translates to a feature of your code and will give you outlines in order to understand better how to build your code. If I was building an recipe list application for example I could use the following to define a service/feature for the application based on the perspective of the user: **As a user, I want to add measurement to the ingredients, so I have a more accuracy for my meals**

To dive deeper into this, check out: [user stories](https://www.atlassian.com/agile/project-management/user-stories#:~:text=Summary%3A%20A%20user%20story%20is,simply%20put%2C%20software%20system%20requirements).

### 6. Schedule Stand-ups

Communication within a team is vital. As part of the agile process, each team must agree on a daily meeting to discuss their daily tasks. Deeper discussions should be held in a separate meeting.

-   Stand-ups are a daily occurrence
-   Stand-ups are meant to be short. Each team member should speak for **no more than 5 minutes**.
-   Team members should talk about what their current tasks; they can also talk about what they did on the previous day (if relevant)
-   Stand-ups are a good time to talk about any issues (blockers) that can prevent a task from moving forward, and if possible, enlist help from other team members
